## Walkthrough-2 — Triage (PL findings, verified against the code 27/06/26)

**T1 — the Tooltip already clamps horizontally but still clips.** `frontend/src/components/Tooltip.tsx` already sets `maxWidth: 280` (`:92`) and clamps `left` to the viewport (`:44-45`). The residual clip is that (a) the bubble's width is only measured **once** on the open frame via a single `requestAnimationFrame` (`:53`) and is never re-clamped on scroll/resize; (b) there is no `max-width` that is viewport-relative, so on a narrow viewport a 280px bubble next to a far-right `.titlebar-meta` InfoTip can still be wider than the space and the clamp pushes `left` negative; (c) there is no vertical (bottom-edge) clamp — only a top flip (`:40-42`). Item 1's tooltip fix is **global** (every `InfoTip`/`Tooltip` consumer) and lands first in PR-B. **Fix shape:** keep the component API identical; make `maxWidth` responsive (`min(280, viewport - 2·MARGIN)`), clamp both axes after measure, and re-run `reposition` on `scroll`/`resize` while open.

**T2 — the row-divider pattern is `borderBottom: var(--border)` on each row inside a padded `.window`.** Confirmed usages: Analytics `OverdueExposure`/`ByFormType` use `borderTop: var(--border)` per `<li>` with `padding: '10px 18px'` (`Analytics.tsx:123,349`); the Filing records list uses `borderBottom: var(--border)` per row with `padding: '12px 18px'` (`FilingStudio.tsx:240`); the Audit filing-picker rows use `borderBottom: var(--border)` with `padding: '14px 18px'` (`AuditDefense.tsx:402`); `ComputationPanel` section bands + hero use `borderBottom: var(--border)` (`FilingPipeline.tsx:230,266,291`); the `.req-list`/`FigureTraceRow` rows; Entity-snapshot field rows (`Entity.tsx`). Because the border sits **inside** the row's horizontal padding, dividers **do not reach** the `.window` left/right outline, and the **last** row's `borderBottom` doubles up with the window's own bottom border. **Fix shape:** a small reusable token-CSS rule (e.g. `.row-divider` / `.row-list > * + *` using a full-bleed top border, OR negative-margin the row's horizontal inset so the border spans full width) + a rule that the **last** child renders no divider. Applied to the enumerated usages only.

**T3 — `filing_records` has no `status`; `FilingRecordReq.computation` is required.** `migrations/neon_schema.sql:61-70` defines `filing_records(id, user_id, tin, label, computation jsonb, risk_flags jsonb, line_items jsonb, created_at)` — **no `status`**. `FilingRecordReq` (`schemas.py:6-11`) has `computation: dict` (**required**) + `risk_flags: list[dict] = []` + `line_items: list[dict] | None = None`. `FilingRepository` (`persistence.py:254-355`) has `create`/`list`/`get`/`delete` — **no update**. The in-memory fallback is `dict[str, list[dict]]`. `create()` always writes in-memory first, then best-effort Neon (`:283-302`). PR-A adds `status`, makes `computation` optional, adds an update/upgrade method + endpoint, and threads `status` through the in-memory + Neon paths + the migration.

**T4 — `/filing/new` result order today is provenance-prose → Computed → Pipeline-is-ABOVE.** In `FilingNew.tsx` the **Filing Pipeline** card renders FIRST (`:513-523`), then the classified-items detail, then on `done` a standalone **provenance `[i]` prose `.window`** (`:574-603`), then `ComputationPanel` "Stage 02-03 - Computed" (`:605`), then the Save/Start-Over buttons (`:621-657`), then `TechnicalDetailsDisclosure` (`:660-662`). Item 4 wants: Pipeline card moved to sit **under** the Computed card, immediately **before** the technical-details disclosure; the provenance prose (`:597-602`) folded into an **InfoTip on the `ComputationPanel` heading**; the manual buttons removed; auto-save on compute. `ComputationPanel` (`FilingPipeline.tsx:211-224`) has a fixed `{title}: {form}` titlebar with **no InfoTip slot** — add an optional `headingTip?: React.ReactNode` prop and render it in the titlebar.

**T6 — the Audit page is already the conversational assistant, single-column.** `AuditDefense.tsx` (Wave-4) already: H1 "Audit Assistant" (`:334`), filing picker (`:341-456`), suggested-question chips seeded from the filing's figures (`seedQuestions`, `:62-86`), chat thread + composer + Send (`:563-646`), per-message `getAuditDefense` (`:282-310`), Trust-Demo fabrication chip (`:541-559`). **What's missing for item 6:** the file/route/nav are still `AuditDefense`/`/audit-defense`/"Audit Defense"; the layout is one column (no LEFT figure pane); and `seedQuestions`/`filingEvidence`/`formatRM` (`:38-86`) render **whatever `computation.fields[*].value` holds** with no sanity bound — a corrupt/overflowing saved record (e.g. `RM 240,000,003,475,967.75`, TIN `Z0000000001`) surfaces verbatim in the chips.
